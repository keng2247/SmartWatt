🐍 Backend — Core App Files

File What It Does (Simple Explanation)
main.py ENTRY POINT The main server file. Starts the FastAPI app, loads all 22 AI models on startup, and sets up CORS so the frontend can talk to the backend.
predictor.py CORE AI The brain of the system. Takes appliance data, runs the correct neural network model, applies the physics formula, calculates monthly kWh, and returns the result with confidence score.
physics_engine.py PHYSICS The wattage calculator. Converts appliance specs (tonnage, star rating, HP) into rated watts using electrical formulas — e.g. AC: tonnage × 1200 × (1 + (5−star)×0.1).
anomaly_engine.py DETECTION The fault detector. Checks if usage hours or efficiency factor are abnormal and raises warnings — Usage Anomaly (too many hours) or Efficiency Critical (25% more power than rated).
kseb_tariff.py BILLING The KSEB bill calculator. Uses 2024-25 telescopic slab rates (₹3.25 to ₹8.80/unit) plus Fuel Surcharge (₹0.13/unit) to calculate the actual bi-monthly electricity bill.
tariff_config.py CONFIG The tariff data store. Stores the KSEB slab rates and FSM value as Python lists — kept separate so you only need to update one file when rates change.
simulation_service.py SIMULATION The "What-If" engine. Runs 8 upgrade scenarios (upgrade AC star rating, switch to BLDC fan, replace old fridge, etc.) and shows how many kWh you would save each month.
schemas.py VALIDATION The input validator. Defines Pydantic models for every appliance — enforces rules like AC tonnage must be 0.5–5.0, star rating 1–5, season must be "summer/monsoon/winter". If wrong → AI never runs.
range_resolver.py UTILITY The range converter. Converts string ranges like "3-5 years" or "10+" into real numbers the AI model can use as input features.
physics_constants.py CONSTANTS The appliance reference values. Stores standard wattage constants for all 22 appliances (e.g., Standard Fan = 75W, BLDC Fan = 30W, LED = 9W, Geyser = 2000W).

⚙️ Backend — Services Folder (services/)

File What It Does (Simple Explanation)
__init__.py The package exporter. Imports and exposes all 5 service classes so the router can use them all with just one import line.
input_normalizer.py The input cleaner. Extracts the common household context (season, occupants, location) from the batch request once and organises each appliance's data separately before prediction.
batch_predictor.py The batch runner. Loops through all selected appliances, calls the predictor for each one, and collects all results into a single list to be returned together.
bias_adjuster.py The self-correction layer. Applies a learned bias multiplier from user_biases.json to fine-tune predictions based on what this user's past data showed.
learning_pipeline.py The auto-retrainer. Every 6 hours, compares predicted kWh vs actual bill. If the new model has a lower MAE and higher R², it replaces the current production model automatically.
system_load_balancer.py The balance keeper. Caps unaccounted electricity (standby, wiring losses) at 15% of total consumption. If exceeded, redistributes excess to major appliances: Fridge 35%, Fan 30%, LED 20%, AC 10%, TV 5%.

🛣️ Backend — Routers Folder (routers/)

File What It Does (Simple Explanation)
appliances.py ALL ENDPOINTS Contains all API endpoints: /predict-appliance (single appliance), /predict-all (full batch with balancing + learning), and /simulate-savings (what-if scenarios). Handles validation, field mapping, and calls the predictor.

🔧 Backend — Utils Folder (utils/)

File What It Does (Simple Explanation)
prediction_logger.py The console logger. Prints a clean, formatted summary of each prediction to the server terminal — shows the source (AI / Physics / Fallback), hours, kWh, and confidence.

💾 Backend — Data & Config Files

File What It Does (Simple Explanation)
learned_params.json The learned adjustment store. Stores the bias correction values per user, updated by the LearningPipeline after every retraining cycle.
user_biases.json The user-specific biases file. Keeps per-user efficiency multipliers that are applied to predictions to personalise the accuracy of results.
.env The secret keys file. Stores the Supabase URL and API key — never shared publicly, loaded by the server at runtime using python-dotenv.
requirements.txt The Python dependency list. Lists all libraries needed (FastAPI, TensorFlow, Pandas, scikit-learn, etc.) — run pip install -r requirements.txt to install everything.
Procfile The deployment instruction. Tells Render (the cloud hosting platform) exactly how to start the server: uvicorn main:app --host 0.0.0.0.
start_backend.bat The local start shortcut. A Windows batch file — double-click it to start the backend server instantly without typing commands.

🤖 Backend — Training & Model Files

File What It Does (Simple Explanation)
train.py ML TRAINING The AI model trainer. Trains all 22 neural network models on the synthetic dataset. Each model predicts Efficiency Factor and Effective Hours. Saves output as models/appliancename_model.keras.
newdataset.py DATA GEN The dataset generator. Uses Monte Carlo simulation to create 12,000 synthetic Kerala household profiles with realistic appliance specs, seasons, and usage patterns for training.
models/ (folder) 22 MODELS The trained AI models store. Contains all 22 .keras neural network model files AND 22 .pkl preprocessor files — one pair per appliance type. Loaded at server startup.
kerala_smartwatt_ai.csv The raw training dataset. The 12,000-row CSV file generated by newdataset.py — used by train.py to train all 22 models.

⚛️ Frontend — App Pages (src/app/)

File What It Does (Simple Explanation)
app/page.tsx MAIN PAGE The 4-step wizard — the heart of the UI. Users enter household info (Step 1), select appliances (Step 2), enter specs (Step 3), and view results with charts and bill estimate (Step 4).
app/layout.tsx The root wrapper. Wraps every page with common HTML structure, Google Fonts, and metadata (title and description for SEO and browser tab).
app/globals.css The global stylesheet. Defines the design system — colours, fonts, animations, and base styles that are shared across the entire application.
app/dashboard/page.tsx The user dashboard. Shows the logged-in user's saved electricity analysis — displays past predictions and history from Supabase.
app/login/page.tsx The login page. Supabase-powered sign-in form for existing users to access their saved analysis.
app/register/page.tsx The registration page. Sign-up form for new users to create an account using Supabase Auth.

🗂️ Frontend — Config Files (src/config/)

File What It Does (Simple Explanation)
appliances.ts The appliance master list. Defines all 22 appliances with their display names, icons, and category groupings shown on the appliance selection screen (Step 2).
usageForms.ts The form field definitions. Specifies exactly which input fields appear for each appliance (e.g., AC needs tonnage + star rating + daily hours). This drives the dynamic input form in Step 3.

📚 Frontend — Library Files (src/lib/)

File What It Does (Simple Explanation)
api/client.ts API The backend connector. Sets up the Axios HTTP client pointing to http://localhost:8000 (or deployed URL) with a 30-second timeout and JSON content type headers.
api.ts The API shortcut. A simple re-export of the API client, making it easy to import and call backend endpoints from any component in the app.
supabaseClient.ts The database connector. Creates and exports the Supabase client instance using the project URL and anon key from .env.local — used for auth and data storage.
authUtils.ts The login helper. Utility functions for checking login status, getting the current user session, and redirecting to login page if not authenticated.
transformFields.ts KEY FILE The field name mapper. Converts UI field names (like ac_star, fridge_capacity) to the exact backend field names (like ac_star_rating, fridge_capacity_liters) before sending to API.
tariffUtils.ts The frontend bill calculator. A TypeScript copy of the KSEB tariff logic so the UI can show a live bill estimate without calling the backend — used for instant feedback.
energyUtils.ts PHYSICS FE The frontend physics engine. Calculates approximate kWh in the browser using physics formulas (same logic as backend's physics_engine.py) — used as a preview before AI results arrive. Also handles the "distribute energy gap" logic to make the total match the user's bill.
normalizePattern.ts The pattern converter. Converts user-selected usage patterns (like "heavy", "moderate") into the exact format the backend expects for AI processing.
physicsConstants.ts The frontend constants store. Contains reference wattage values (Fan=75W, LED=9W, BLDC=30W, etc.) and physics ratios used by energyUtils.ts for browser-side calculations.
types.ts The TypeScript type definitions. Defines shared interfaces like ApplianceUsageDetails, PredictionResult, and HouseholdData used across all frontend files for type safety.
usageAlerts.ts The alert message generator. Checks prediction results and creates human-readable tips (e.g., "Your geyser runs too long — consider reducing to 30 min/day") shown on the results screen.
generatePDF.ts The PDF report creator. Uses jsPDF to generate a downloadable PDF report of the full electricity analysis including all appliance data, charts summary, and KSEB bill estimate.
diagnostics.tsx The developer debug panel. A hidden component that shows API connection status, backend health check result, and model loading state — only used during development and testing.
verifyPersistence.ts The save verifier. After saving user data to Supabase, this checks that the data was actually written successfully and logs any database errors to the console.
utils.ts The general utilities file. Contains small helper functions used throughout the frontend — mainly the cn() function for merging Tailwind CSS class names conditionally.

⚙️ Frontend — Setup & Config Files

File What It Does (Simple Explanation)
next.config.ts The Next.js configuration. Project-level settings for the framework — image optimization domains, environment variable exposure, and build options.
package.json The frontend dependency list. Lists all JavaScript libraries (Next.js, React, Axios, Recharts, jsPDF, Supabase, Plotly, etc.) and defines npm scripts like npm run dev.
tsconfig.json The TypeScript compiler settings. Enables strict mode and sets the @/ path alias to point to the src/ folder so imports are shorter and cleaner.
eslint.config.mjs The code quality rules. ESLint configuration that enforces consistent coding standards across the frontend — we verified zero warnings during our audit.
.env.local The frontend secrets file. Stores NEXT_PUBLIC_SUPABASE_URL and NEXT_PUBLIC_SUPABASE_ANON_KEY — needed for database and auth to work.
.env.example The environment template. Shows all variables that need to be configured — shared safely as a reference without exposing real keys.
vercel.json The Vercel deployment config. Tells Vercel (the frontend cloud host) how to build and serve the Next.js application.
